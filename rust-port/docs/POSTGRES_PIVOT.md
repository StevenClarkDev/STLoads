# PostgreSQL Pivot

This document tracks the database pivot for the Rust port.

## Target

- Runtime target: IBM-hosted Rust services
- Relational database target: PostgreSQL
- Current production source: Laravel + MySQL

## Important Reality

The Rust workspace now compiles against SQLx PostgreSQL types and pools.
That is a meaningful milestone, but the repository is not production-ready on PostgreSQL yet.
The remaining work is in raw SQL text, migrations, and integration-test fixtures.

## PostgreSQL Progress Completed

- `rust-port/Cargo.toml`
  - SQLx now compiles with the `postgres` feature.
- `rust-port/crates/db/src/pool.rs`
  - now uses `PgPool` and `PgPoolOptions`.
- `rust-port/crates/db/src/tms.rs`
  - transaction helpers now use `sqlx::Transaction<'_, sqlx::Postgres>`.
- Generated-id insert paths in:
  - `rust-port/crates/db/src/auth.rs`
  - `rust-port/crates/db/src/marketplace.rs`
  - `rust-port/crates/db/src/tms.rs`
  - now use `RETURNING id` instead of MySQL `last_insert_id()` access.

## Remaining MySQL-Shaped Hotspots

- `rust-port/crates/db/migrations/*.sql`
  - still use MySQL DDL such as `AUTO_INCREMENT`, `DATETIME(6)`, `ENGINE=InnoDB`, `utf8mb4`, and MySQL foreign-key/index syntax.
- `rust-port/crates/db/src/*.rs`
  - many queries still use MySQL-specific placeholders and expressions such as:
    - `?` bind placeholders instead of PostgreSQL `$1`, `$2`, ...
    - `UTC_TIMESTAMP(6)`
    - `DATE_SUB(...)`
    - `DATE_ADD(...)`
    - `ON DUPLICATE KEY UPDATE`
- `rust-port/crates/db/tests/lifecycle_integration.rs`
  - test fixture SQL is still written in MySQL-flavored syntax.

## PostgreSQL Port Tasks

1. Rewrite migrations `0001` through `0005` into PostgreSQL DDL.
2. Replace MySQL placeholder and function assumptions throughout repository code.
3. Convert MySQL upsert syntax to PostgreSQL `ON CONFLICT ... DO UPDATE`.
4. Re-run and fix the DB integration tests against a disposable PostgreSQL database.
5. Only after the above, point IBM deployment `DATABASE_URL` at the target PostgreSQL service for live persistence.

## SQL Translation Notes

Typical SQL changes we should expect:

- `AUTO_INCREMENT` -> `GENERATED BY DEFAULT AS IDENTITY` or `BIGSERIAL`
- `DATETIME(6)` -> `TIMESTAMPTZ` or `TIMESTAMP`
- `UTC_TIMESTAMP(6)` -> `CURRENT_TIMESTAMP`
- `DATE_SUB(UTC_TIMESTAMP(6), INTERVAL 30 DAY)` -> `CURRENT_TIMESTAMP - INTERVAL '30 days'`
- `DATE_ADD(UTC_TIMESTAMP(6), INTERVAL 1 DAY)` -> `CURRENT_TIMESTAMP + INTERVAL '1 day'`
- MySQL JSON handling -> PostgreSQL `JSONB`
- `ON DUPLICATE KEY UPDATE` -> `ON CONFLICT (...) DO UPDATE`
- MySQL engine/charset clauses -> removed

## Rule For Ongoing Work

Every new DB-facing feature should be written so the PostgreSQL pivot gets easier, not harder.
That means:

- avoid introducing new MySQL-only SQL when it can be written portably
- keep IBM deployment assumptions env-driven
- prefer clear schema and query documentation when a shortcut would lock us deeper into MySQL
